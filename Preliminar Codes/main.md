- UART Communication Variables:

1. Single‑byte receive buffer: uint8_t rx_byte; // A temporary variable where the UART interrupt stores the latest received byte.

Note: UART hardware delivers one byte at a time, so the interrupt handler reads that byte into rx_byte. The interrupt cannot process full commands. It only grabs the byte and passes it to your parser.

<img width="440" height="38" alt="image" src="https://github.com/user-attachments/assets/8fb471c4-0970-48dc-9eae-5f975196bae2" />

2. Temporary character buffer (accumulates characters): char rx_buffer[64]; // A 64‑character buffer where you accumulate incoming characters until a full command is received. Size 64 because it is enough space for typical commands. Prevents overflow.

- <img width="298" height="23" alt="image" src="https://github.com/user-attachments/assets/af68afa1-7c3c-4912-8bdb-9ec09936dabb" />

- <img width="196" height="241" alt="image" src="https://github.com/user-attachments/assets/e76b6ad5-b922-42b4-babe-52291fcd1c56" />

3. Final command buffer: char command[64]; // When a newline \n arrives, the interrupt copies the full command from rx_buffer into command.

Note: Command becomes the stable, ready‑to‑process string. Meanwhile, rx_buffer can start receiving the next command.

4. Index inside rx_buffer: uint8_t rx_index = 0; // Tracks where the next incoming character should be stored inside rx_buffer.

- <img width="458" height="26" alt="image" src="https://github.com/user-attachments/assets/aa81d67d-2ede-4474-ae52-6de061867f23" />

5. Command‑ready flag: volatile uint8_t command_ready = 0; // A flag set by the UART interrupt: 0 → no complete command yet, 1 → a full command (ending in '\n') is ready.

Note: It is volatile because it is modified inside an interrupt. The main loop must always read the latest value. It is used in the while loop, everytime a command is ready the variable is cleaned (set to 0) and the function process command is called.

- Private variables (HAL peripheral handles): These are HAL driver objects that represent your UART peripherals. Each handle stores: Baud rate, word length, stop bits, parity, hardware registers, DMA settings, Interrupt state. Every HAL function needs a handle to know which UART we’re talking to. Without these handles, the HAL cannot operate the UART hardware.

huart1 → configuration + state for USART1 (Used for ESP COM)
huart2 → configuration + state for USART2 (Used for Debug, serial monitor,...)

<img width="164" height="42" alt="image" src="https://github.com/user-attachments/assets/a05ce43e-3f8b-4ce6-8eb4-895316297b74" />

- Private function prototypes (HAL system functions):These are function prototypes for hardware initialization routines.

1. SystemClock_Config() - Configures: PLL, HCLK, PCLK1, PCLK2, Flash latency, Oscillator sourceThis sets the CPU frequency and all peripheral clocks.

2. MX_GPIO_Init() - Initializes all GPIO pins: Modes (input/output/AF/analog), Pull‑ups / pull‑downs, Output type, Speed.

3. MX_USART1_UART_Init() - Configures UART1: Baud rate, Word length, Stop bits, Parity, Mode (TX/RX), Interrupts.

4. MX_USART2_UART_Init() - Same as above, but for UART2.

<img width="218" height="48" alt="image" src="https://github.com/user-attachments/assets/04b39cf7-44b6-4ec7-86b9-c584721278a1" />

- USER CODE BEGIN PFP / END PFP: region where you declare your own function prototypes. CubeMX will not touch anything inside. These functions are part of the application logic, not HAL.

1. Process_Command() - Called when command_ready == 1. Parses the received command string. Executes actions like: Turn LED on/off, Move servo, Read temperature, Control fan.

2. Send_Status() - Sends a formatted status message back to the ESP8266. Includes: Temperature, Humidity, Motion state, Fan speed, LED state.

MAIN FUNCTION - int main(void)****

- Reset HAL and core system. Resets all HAL state and low‑level drivers. Initializes: Flash interface (wait states, access). SysTick timer (used for HAL_Delay() and timebase)

<img width="454" height="27" alt="image" src="https://github.com/user-attachments/assets/e8b8e745-8ad3-4ec2-bd4e-924ee0ccf04c" />

- System clock configuration: Sets up: Oscillator source (HSI, HSE, etc.), PLL multipliers/dividers, Core clock (HCLK), Peripheral clocks (PCLK1, PCLK2), This defines the CPU speed and all timing for peripherals (UART baud, timers, etc.). Your delay_us() and UART baud rates depend on this being correct.

<img width="194" height="26" alt="image" src="https://github.com/user-attachments/assets/ebb5f7dd-1008-4ffd-9c35-2700e332b8bf" />

- Initialize all configured peripherals (HAL/CubeMX): These functions are auto‑generated by CubeMX and use HAL internally.

1. MX_GPIO_Init(). Configures all GPIO pins: Modes (input, output, alternate, analog), Pull‑ups/pull‑downs, Output type (push‑pull/open‑drain), Speed. This is where PC1, PC2, PC3, etc. get their basic configuration.

2. MX_USART1_UART_Init(). Sets up UART1: Baud rate, Word length, Stop bits, Parity, TX/RX mode, Used for communication with the ESP8266 (in your case).

2. MX_USART2_UART_Init(). Same idea, but for UART2. Often used for debugging via USB‑serial or another external device.

- Start UART interrupt reception: Enables interrupt‑based reception on UART1.

Parameters: 
1. &huart1 → which UART (USART1)
2. &rx_byte → where to store the received byte
3. 1 → receive one byte at a time

After this call: 
1. When a byte arrives from ESP8266, the UART interrupt fires.
2. HAL calls your HAL_UART_RxCpltCallback().
3. You append rx_byte into rx_buffer, update rx_index, and possibly set command_ready.

<img width="482" height="29" alt="image" src="https://github.com/user-attachments/assets/1c8ac0ea-862c-416b-8279-1060acc16615" />

WHILE****

- Process command from ESP8266: command_ready is set by the UART interrupt when a full command (ending in \n) is received. If it’s 1, you: Clear it back to 0 so the same command isn’t processed twice. Call Process_Command(), which reads the global command buffer and: Opens/closes door, Turns lights on/off or auto, Changes fan mode, etc.

<img width="265" height="70" alt="image" src="https://github.com/user-attachments/assets/82bcb48a-e57e-4c93-aa61-1ba77b1f2604" />

- Update DHT + fan PID control - This function typically: Reads temperature (via DHT_Read() inside or before). Updates the PID controller. Adjusts the fan PWM (PB6) based on temperature and setpoint. It’s called every loop, so the fan reacts continuously to temperature changes.

<img width="200" height="29" alt="image" src="https://github.com/user-attachments/assets/6f6c070e-6d92-464e-8d03-34cb27b2f091" />

- Check PIR every 200 ms: HAL_GetTick() returns milliseconds since startup. last_pir_time stores the last time you checked the PIR. Every 200 ms: You update last_pir_time. Call PIR_RawRead() to read the motion sensor (PC2). Store result in motion_detected (likely 0/1).
